## 1. 系統定位與邊界

### 1.1 產品定位

本系統是一個 **多空強度預測指標（Market Intelligence Indicator）**，預測 BTC 永續合約未來 4 小時的方向與幅度，以圖表形式可視化並透過 Telegram 推送。

**明確不是**：
- ❌ 交易策略或自動下單系統
- ❌ 進出場規則 / TP / SL / 倉位管理 / 槓桿控制
- ❌ 交易績效最大化或回測框架
- ❌ 執行層（fee simulation、order routing 等）

**評估以預測品質為準**：IC、ICIR、方向準確率、calibration monotonicity、Strong 訊號勝率。**不做交易績效回測**。

### 1.2 核心輸出（每小時整點產出一次）

| 欄位 | 型別 | 說明 |
|------|------|------|
| `pred_return_4h` | float | sign(direction) × magnitude（預期 4h 收益） |
| `pred_direction` | enum | `UP` / `DOWN` / `NEUTRAL` |
| `strength_score` | enum | `Strong` / `Moderate` / `Weak` |
| `confidence_score` | 0–100 | 複合信心分數（見 §8.1） |
| `mag_pred` | float | |return_4h| 預測幅度 |
| `dir_prob_up` | 0–1 | P(UP) 原始機率 |
| `bull_bear_power` | -1 至 +1 | 跨源情緒綜合指標 |
| `regime` | string | `CHOPPY` / `TRENDING_BULL` / `TRENDING_BEAR` / `WARMUP` |

### 1.3 核心 target

```python
y_return_4h = close.shift(-4) / close - 1
```

所有特徵計算均為 **trailing-only**（無前視偏差）。`build_live_features()` 同時供訓練與生產使用，保證 train/serve 一致性。

### 1.4 系統規模現狀

| 項目 | 數值 |
|------|------|
| 資料量 | ~4245 bars 1h K 線（168 天） |
| 總特徵數 | 130+ 工程特徵（12 群組） |
| Direction 生產特徵 | 29（pruned）/ 34（ablation-validated，待 retrain） |
| Magnitude 生產特徵 | 87 |
| 模型數 | 2（Direction + Magnitude，各 1 個 XGBoost） |
| 資料源 | Binance REST、Coinglass API v4（24 個端點）、Deribit Public API（2 個端點） |
| 預測視窗 | 4 小時（4 × 1h bars） |
| 更新頻率 | 每小時整點（:02 cron 觸發） |

### 3.1 Binance 資料

|----------|------|------|
| `/fapi/v1/klines` | 1h OHLCV + taker buy volume | 每小時 500 bars |

### 3.2 Coinglass API v4

**24 個 endpoints**，分兩類：

**時序類（15 個）**— 原生 1h：
- OI：`oi_history`, `oi_aggregated_history`
- 清算：`liquidation_history`
- Long/Short：`global_ls_account`, `ls_ratio`, `top_trader_account_ls`
- Funding：`funding_rate_history`, `oi_weighted_funding_history`
- Taker：`taker_buysell_volume`
- 額外指標：`coinbase_premium`, `bitfinex_margin`, `binance_spot_cvd`, `liquidation_heatmap`, `large_orders_history`, `rsi_history`

**快照類（9 個）**— 即時值（只寫最後一根 bar）：
- `whale_index`, `fear_greed`, `puell_multiple`, `ahr999`, `pi_cycle`, `rainbow`, `stablecoin_marketcap`, `bubble_index`, `sentiment_index`

**關鍵設計**：使用 `merge_asof(direction='backward')` 精確對齊到 1h bar，**不做 forward-fill**（避免偽訊號）。

### 3.3 Deribit Public API

**2 個 endpoints**（`indicator/data_fetcher.py` → `fetch_deribit_*`）：
- `public/get_historical_volatility`：DVOL 波動率指數
- `public/get_book_summary_by_currency`：選擇權 summary（IV、open interest per strike）

### 3.4 Timestamp 格式陷阱（歷史教訓）

**不同 endpoint 可能用不同 unit**：Coinglass 大部分 endpoint 用 13 位 ms，但 `coinbase_premium` 和 `bitfinex_margin` 用 10 位 s。解析時必須自動偵測：

```python
unit = "s" if sample_ts < 1e12 else "ms"
```

2026-04-12 的事故中，硬編碼 `unit="ms"` 導致秒級時間戳被解析成 1970 年，造成 5.5 天資料缺口。詳見 `.claude/rules/mistake.md`。

### 5.2 Z-Score 正規化

所有 Coinglass 特徵都有對應的 `_zscore` 變體：

```python
zscore = (value - rolling_mean_24h) / rolling_std_24h
```

- 窗口：24 bars（24h）
- BBP 使用時截斷至 [-3, 3]

### 5.3 Interaction 特徵設計規則（血淋淋的教訓）

**禁止寫 `feature × sparse_indicator`**（2026-04-13 教訓）：當 indicator 非零比例 < 30%，乘出來的特徵會被 sparsity pattern 主導，跟其他用同一 indicator 乘的特徵高度相關（>0.95），IC 反而 collapse。

**正確寫法**：
- `feature × (1 - is_dead_regime)` — 只在 base feature 在某 regime 完全失效時屏蔽噪音
- 把 indicator 本身當 feature 加進去，讓 XGB 自己做 tree split 決定 conditional rule

**強制步驟**：設計完任何 interaction feature，必須跑 inter-feature correlation matrix，如果一群「本應獨立」的特徵互相相關 > 0.9，那不是訊號，是 indicator pattern leakage。

### 5.4 特徵選擇

**目前生產（Direction model）**：
1. 從 88 個候選特徵出發
2. 透過 `research/auto_pruner.py` 用 IC-filter + VIF redundancy removal 剪枝
3. 結果：29 特徵（生產中）

**Ablation-validated（待下次 retrain 上線）**：
1. Baseline = 29 特徵 pruned set
2. 用 `research/ablation_study.py` 做 forward selection，逐一評估 6 個新候選特徵
3. 每個候選看 walk-forward OOS AUC delta（相對 baseline）
4. KEEP 條件：delta_auc ≥ 0.001 且不傷 train/test 對稱性
5. 結果：5/6 KEEP → combined 34 特徵，OOS AUC 0.5953 → 0.6020

## 6. Dual Model ML 管線

### 6.1 架構概覽

**v7 Dual Model** 將方向與幅度預測拆分為**兩條完全獨立**的 XGBoost pipeline：

```
┌─────────────────────────┐      ┌─────────────────────────┐
│ Direction Model         │      │ Magnitude Model         │
│                         │      │                         │
│ XGBClassifier           │      │ XGBRegressor            │
│ target: y_dir (0/1)     │      │ target: |return_4h|     │
│ features: 29 (prod)     │      │ features: 87            │
│ output: P(UP)           │      │ output: mag_pred        │
│                         │      │                         │
│ build_direction_labels  │      │ build_magnitude_labels  │
└─────────────────────────┘      └─────────────────────────┘
            │                                 │
            └────────────┬────────────────────┘
                         ▼
                 最終複合訊號生成
                 （見 §8）
```

### 6.2 Direction Classifier

**Target**：`y_dir = (close.shift(-4) / close - 1 > k × sigma).astype(int)`，其中 `k=0.5`、`sigma` 是 rolling realized vol。死區內的樣本 label 為 NaN（不訓練、不算 loss）。

**XGBoost 參數**（生產）：
```python
DIR_PARAMS = {
    "objective": "binary:logistic",
    "eval_metric": "auc",
    "max_depth": 4,
    "learning_rate": 0.05,
    "n_estimators": 200,
    "subsample": 0.8,
    "colsample_bytree": 0.7,
    "min_child_weight": 10,
    "reg_alpha": 0.1,
    "reg_lambda": 1.0,
    "random_state": 42,
}
```

**Regime weighting**（2026-04-09 上線）：
```python
weights = np.ones(len(df))
weights[bull_mask] = 4.0   # TRENDING_BULL 樣本權重 4x
weights[bear_mask] = 2.0   # TRENDING_BEAR 樣本權重 2x
```

原因：BULL regime 在資料中只佔 ~15%，但是生產關心的主要場景，用 sample_weight 補償。

**Class imbalance**：
```python
scale_pos_weight = (1 - up_rate) / up_rate
```

### 6.3 Magnitude Regressor

**Target**：`|close.shift(-4) / close - 1|`，σ 正規化後訓練，推論時乘回 σ 還原到原始單位（2026-04-12 修過一個 σ-scale mismatch bug）。

**特色 alpha 特徵**（均經 IC 驗證）：
- `impact_asymmetry`：IC = -0.071（買賣衝擊失衡）
- `post_absorb_breakout`：mag IC = 0.191（吸收後突破，magnitude 第一重要特徵）
- Liquidity fragility features（`c02748e` 加入）

### 6.4 Walk-Forward 交叉驗證

```python
walk_forward_splits(
    n=len(df),
    initial_train=288,  # 12 天暖機
    test_size=48,       # 每折測 48 小時 = 2 天
    step=48,            # 每 2 天滑動一次
)
# 結果：77 折
```

- 擴展窗口（訓練始終從 bar 0 開始）
- 每折獨立訓練，OOS 預測永遠不用於任何訓練決策
- 嚴禁資訊洩漏

### 6.5 Target Label 建構

```python
# research/dual_model/build_direction_labels.py
def build_direction_labels(df, k=0.5):
    returns = df['close'].shift(-4) / df['close'] - 1
    sigma = log_returns.rolling(20).std() * np.sqrt(4)
    threshold = k * sigma
    y_dir = np.where(returns > threshold, 1,
                     np.where(returns < -threshold, 0, np.nan))
    return {'y_dir': y_dir, 'return_4h': returns}
```

死區樣本（|return| < k × σ）不參與訓練，避免模型在噪音區學到無意義的 pattern。

## 8. 訊號生成與 Gating

### 8.1 Confidence 分數

```python
# mag_percentile: 當前 mag_pred 在 pred_history 中的擴展百分位
# dir_conviction: |dir_prob_up - 0.5| × 2（0=中性，1=極確定）
confidence_score = mag_percentile × (0.7 + 0.3 × dir_conviction) × 100
```

**強度分級**：
- `Strong`：confidence ≥ 80
- `Moderate`：confidence ≥ 65
- `Weak`：confidence < 65

### 8.2 方向判定

```python
if dir_prob_up > 0.60:   pred_direction = "UP"
elif dir_prob_up < 0.40: pred_direction = "DOWN"
else:                    pred_direction = "NEUTRAL"
```

### 8.3 四層 Gating

| Gate | 邏輯 | 目的 |
|------|------|------|
| **BBP 確認閘門** | Bull/Bear Power 方向必須與 pred_direction 一致 | 拒絕與跨源情緒矛盾的訊號 |
| **Regime 動態死區** | CHOPPY regime deadzone 更寬（0.50→0.55），TRENDING 更窄（0.45） | 減少 CHOPPY 噪音，放行 TRENDING 確信訊號 |
| **Hysteresis** | 連續 2 根 bar 同向才確認 flip | 防止抖動 |
| **Cooldown** | flip 後強制 1 根 bar 冷卻 | 避免頻繁換邊 |

### 8.4 Strong 訊號特別處理

當 `strength_score == "Strong"` 時額外觸發：
1. **SHAP driver 分析**：用 `SignalExplainer`（SHAP TreeExplainer）算 top 5 推動因子
2. **Telegram 文字告警**：推 structured message 含驅動因子
3. **signal_tracker 記錄**：寫入 `tracked_signals` 表，預設 4h 後由 `backfill_outcomes()` 自動回填 actual_return_4h 和 correct
4. **inline keyboard**：提供 /chart、/ichart、/perf 快捷按鈕

## 11. 模型生命週期與 Retrain 流程

### 11.1 觸發條件

**定期**：資料累積到 >= 5500 bars（約每 2-3 週一次）
**緊急**：偵測到
- Rolling IC 持續 7 天負值
- Train/serve diff 有 critical feature drift
- Feature importance tracker 檢測到 top-10 overlap < 7

### 11.2 完整流程

1. **Pre-flight**：`bash retrain_pipeline.sh`（dry run）驗證所有 research artifact
2. **Code review**：檢查 `results/*.json`：
   - ablation：哪些 KEEP、哪些 DROP
   - permutation：p_auc < 0.05 AND p_ic < 0.05
   - train_serve_diff：零 drift
3. **Deploy**：`bash retrain_pipeline.sh --deploy`
4. **Post-deploy**：
   - 重啟 indicator service 載入新模型
   - 監控 24h `/perf` 輸出
   - 觀察前 5 個 Strong 訊號是否正常
5. **Commit & push**：
   ```
   git add indicator/model_artifacts/dual_model/
   git commit -m "Retrain dual model (ablation-validated, Npresamples)"
   git push origin main
   ```

### 11.3 Rollback 機制

每次 retrain 自動備份到 `dual_model_backup_YYYYMMDD/`。緊急還原：
```bash
cp -r indicator/model_artifacts/dual_model_backup_YYYYMMDD/* \
      indicator/model_artifacts/dual_model/
```

然後重啟 indicator service。

### 11.4 Retrain 後不保證改進

**血淋淋的教訓**（2026-04-13）：不是每次 retrain 都會讓模型變好。
- 增加特徵 ≠ 改善（sample:feature ratio 變差會 overfit）
- 必須透過 ablation 驗證每個新特徵
- 必須透過 permutation 驗證 OOS metric 是真 signal 而非 curve fitting
- 必須透過 `model version guard` 評估才不會混淆新舊模型

## 16. 已知問題與關鍵設計決策

### 16.1 核心原則（永遠不能違反）

1. **無前視偏差**：所有特徵計算使用 trailing-only rolling
2. **歷史與即時一致性**：`build_live_features()` 同時供訓練和推論使用
3. **時間對齊精準**：Coinglass 使用 `merge_asof backward` 對齊，快照數據只設定最後一根 bar
4. **預測導向評估**：IC、方向準確率、calibration。**不做交易績效回測**
5. **新特徵先回測**：必須先跑 IC / AUC 驗證才加進系統
6. **圖表同步規則**：靜態與互動圖表必須同步更新
7. **評估前先做 sanity check**：確認樣本在同一模型版本下產生、沒混入資料污染窗

### 16.2 已知問題

| 問題 | 狀態 | 備註 |
|------|------|------|
| Coinglass OI / funding / sentiment + Deribit endpoints 100% NaN | 已知未解（2026-04-12）| 見 `memory/project_coinglass_issue.md` |
| indicator_history 前 6 個月沒 dir_prob_up | 未解 | Dual model 在 2026-04-03 才部署，歷史無法反向填（會是 in-sample）|
| Python 3.9 vs 3.11 版本差異 | 本地 workaround | research 腳本不 import `shared.db`（會因 `Queue \| None` type hint 炸），直接用 pymysql 連 Railway |

### 16.3 關鍵設計決策歷史

| 決策 | 日期 | 原因 |
