## 1. 系統概覽

### 1.1 產品定位

本系統是一個 **多空強度預測指標（Market Intelligence Indicator）** — 預測 BTC 永續合約未來 4 小時的價格方向與幅度。本系統明確**不是**交易策略：沒有進出場規則、沒有倉位管理、沒有止盈止損邏輯、沒有執行層。

### 1.2 核心輸出

每小時整點產出以下預測：

| 欄位 | 型別 | 說明 |
|------|------|------|
| `pred_return_4h` | float | 預測未來 4h 收益率（如 +0.12%） |
| `pred_direction` | enum | `UP` / `DOWN` / `NEUTRAL` |
| `confidence_score` | 0–100 | 校準後的預測信心分數 |
| `strength_score` | enum | `Strong` / `Moderate` / `Weak` |
| `bull_bear_power` | -1 至 +1 | 多源情緒綜合指標 |
| `regime` | string | 當前市場狀態分類 |

### 1.3 系統規模

- **資料量**：約 168 天的 1h K 線（4,006 根），72 個基礎特徵（含新增 OI 衍生特徵）
- **模型特徵**：69–71 個（per-target feature selection，每個 target 獨立篩選）
- **模型數量**：12 個 XGBoost 模型（2 active targets × (1 global + 3 regime-specific)）
- **預測視窗**：4 小時（4 × 1h bars）
- **更新頻率**：每小時整點（:02 觸發）
- **資料來源**：Binance aggTrades WebSocket、OKX WebSocket、Binance REST（K 線）、Coinglass API v4（7 個端點）

## 3. 資料來源與接收層

### 3.1 交易所適配器

系統透過持久 WebSocket 連線，從兩大衍生品交易所接收 tick 級成交資料：

**Binance Futures 適配器** (`market_data/adapters/binance_trades.py`)
- **協議**：WebSocket (`wss://fstream.binance.com/ws`)
- **串流**：aggTrade（聚合成交 — 同價位的多筆成交合併）
- **資料**：價格、數量（base 單位）、taker 方向、成交 ID、交易所時戳
- **合約**：線性永續（BTCUSDT），contract_size = 1（base 單位）

**OKX Futures 適配器** (`market_data/adapters/okx_trades.py`)
- **協議**：WebSocket (`wss://ws.okx.com:8443/ws/v5/public`)
- **串流**：逐筆成交（非聚合）
- **資料**：價格、數量（合約張數）、方向、成交 ID、交易所時戳
- **合約**：反向式記帳，contract_size = 0.01 BTC（BTC-USDT-SWAP）

### 3.2 成交正規化 (`market_data/core/trade_normalizer.py`)

所有原始成交經統一正規化管線：

```
交易所原始成交 → normalize() → 統一成交格式
```

**統一成交格式**：
```python
{
    "exchange": "binance",               # 來源交易所
    "raw_symbol": "BTCUSDT",            # 交易所原生代碼
    "canonical_symbol": "BTC-USD",       # 統一內部代碼
    "instrument_type": "perp",           # 永續合約
    "price": 84250.5,                    # 成交價格
    "size": 0.5,                         # 原生單位數量
    "size_unit": "base",                 # "base" 或 "contract"
    "taker_side": "buy",                 # 主動方方向
    "notional_usd": 42125.25,           # price × size × contract_size
    "trade_id": "123456789",
    "ts_exchange": 1711929600000,        # Unix 毫秒
    "ts_received": 1711929600050,        # Unix 毫秒
    "is_aggregated_trade": True          # Binance aggTrade = True
}
```

**代碼映射** (`market_data/core/symbol_mapper.py`)：
- `BTCUSDT` → `BTC-USD`（Binance）
- `BTC-USDT-SWAP` → `BTC-USD`（OKX）
- 統一格式：`{BASE}-USD`（大寫，連字號分隔）

### 3.3 資金流聚合 (`market_data/core/flow_aggregator.py`)

正規化後的成交累積為 1 分鐘 flow bars：

```python
class FlowAggregator:
    def add_trade(self, trade: dict)      # 執行緒安全累積
    def flush(self, now_ms: int) -> list  # 輸出已完成的 bars
```

**每根 bar 計算欄位**：

| 欄位 | 公式 |
|------|------|
| `buy_notional_usd` | Σ notional（taker_side = "buy"） |
| `sell_notional_usd` | Σ notional（taker_side = "sell"） |
| `delta_usd` | buy - sell |
| `volume_usd` | buy + sell |
| `trade_count` | 成交筆數 |
| `cvd_usd` | cvd(前一根 bar) + delta(當前 bar) |
| `source_count` | 貢獻的交易所數量 |

- **桶鍵**：(canonical_symbol, minute_start_ms)
- **刷出間隔**：約 10 秒，僅輸出已完成的 bar（window_end ≤ now）
- **CVD**：累積制 — 跨 bar 攜帶狀態，維持連續追蹤
- **執行緒安全**：`threading.Lock()` 保護共用累積器狀態

### 3.4 Coinglass API 整合

Coinglass v4 API 的六個端點提供衍生品市場結構資料：

| 端點 | 資料 | 關鍵欄位 |
|------|------|----------|
| `oi` | 各交易所未平倉量 | Binance OI、佔比 |
| `oi_agg` | 全交易所聚合 OI | 總 OI、delta、加速度 |
| `liquidation` | 強制清算量 | 多/空清算 USD |
| `long_short` | 大戶多空比 | 多方%、帳戶比 |
| `global_ls` | 全局帳戶多空比 | 全局多方%、背離值 |
| `funding` | 資金費率 OHLC | 收盤費率、範圍 |
| `taker` | Taker 買賣量 | Delta、買賣比 |

- **原生精度**：1h（避免 forward-fill 造成的偽訊號）
- **歷史回補**：`market_data/backfill/coinglass_backfill.py`（Parquet 儲存）
- **即時抓取**：`indicator/data_fetcher.py` → `fetch_coinglass(interval="1h", limit=500)`

## 5. 特徵工程管線

### 5.1 特徵分類

系統從 4 個資料來源計算 72 個基礎特徵（v4: 含新增 OI 衍生特徵）：

#### A. 價格與成交量特徵（Binance 1h K 線）

| 特徵 | 公式 | 設計邏輯 |
|------|------|----------|
| `log_return` | ln(close / close[-1]) | 基礎收益率序列 |
| `realized_vol_20b` | std(log_return, 20) | 20 根 bar 實現波動率 |
| `return_skew` | skew(log_return, 24) | 收益分佈偏態 |
| `return_kurtosis` | kurtosis(log_return, 24) | 尾部風險度量 |
| `volume_ma_4h` | volume / MA(volume, 4) | 短期成交量異常 |
| `volume_ma_24h` | volume / MA(volume, 24) | 日成交量異常 |
| `return_lag_1..10` | log_return[-1..-10] | 自迴歸分量 |

#### B. Taker 資金流特徵（Binance K 線 + Coinglass）

| 特徵 | 公式 | 設計邏輯 |
|------|------|----------|
| `taker_delta_ratio` | (taker_buy_vol - taker_sell_vol) / total_vol | 每根 bar 買賣失衡率 |
| `taker_delta_ma_4h` | MA(taker_delta_ratio, 4) | 短期資金流趨勢 |
| `taker_delta_ma_24h` | MA(taker_delta_ratio, 24) | 日資金流趨勢 |
| `taker_delta_std_4h` | std(taker_delta_ratio, 4) | 資金流波動性 |
| `taker_delta_std_24h` | std(taker_delta_ratio, 24) | 日資金流穩定性 |

#### C. Coinglass 衍生品特徵（6 個資料源，30+ 特徵）

**未平倉量**（v4 大幅擴充）：
- `cg_oi_close`、`cg_oi_delta`、`cg_oi_accel` — 水位、變化量、加速度
- `cg_oi_close_pctchg_4h/8h/12h/24h` — 多時間框架 OI 變化率（v4 新增）
- `cg_oi_range_zscore`、`cg_oi_range_pct` — OI bar 內振幅（v4 新增）
- `cg_oi_upper_shadow` — OI 上影線 = high - max(open, close)（v4 新增）
- `cg_oi_agg_delta`、`cg_oi_agg_delta_zscore` — 聚合 OI 動量
- `cg_oi_binance_share`、`cg_oi_binance_share_zscore` — Binance 佔比及 z-score（v4 新增）

**清算**：
- `cg_liq_long`、`cg_liq_short` — 多/空清算量
- `cg_liq_ratio`、`cg_liq_imbalance` — 清算方向偏態

**多空比**：
- `cg_ls_ratio`、`cg_ls_long_pct` — 大戶持倉比例
- `cg_gls_ratio` — 全局帳戶比例
- `cg_ls_divergence` — 大戶 vs 全局背離度

**資金費率**：
- `cg_funding_close`、`cg_funding_range` — 當前費率與區間
- `cg_funding_close_zscore` — 標準化資金費率偏差

**Taker 成交量**：
- `cg_taker_delta`、`cg_taker_ratio` — 淨 taker 流量
- `cg_taker_delta_zscore` — 標準化 taker 動量

#### D. 跨指標衍生信號

| 特徵 | 公式 | 設計邏輯 |
|------|------|----------|
| `oi_price_divergence` | oi_delta_zscore - price_return_zscore | OI 與價格背離 → 倉位堆積 |
| `funding_taker_align` | sign(funding) × sign(taker_delta) | 資金費率與 taker 方向一致 = 確信度 |
| `cg_crowding` | abs(ls_ratio_z) + abs(funding_z) + abs(oi_z) | 極端持倉 → 反轉風險 |
| `cg_conviction` | taker_delta_z × oi_delta_z | 資金流與 OI 同向 = 真實流量 |

#### E. 時間編碼

| 特徵 | 公式 | 設計邏輯 |
|------|------|----------|
| `hour_sin` | sin(2π × hour / 24) | 週期性時段效應 |
| `hour_cos` | cos(2π × hour / 24) | 週期性時段效應 |
| `weekday_sin` | sin(2π × weekday / 7) | 星期幾季節性 |
| `weekday_cos` | cos(2π × weekday / 7) | 星期幾季節性 |

### 5.2 Z-Score 正規化

所有 Coinglass 特徵都有對應的 `_zscore` 變體：

```python
zscore = (value - rolling_mean_24h) / rolling_std_24h
```

- **窗口**：24 bars（1h 資料下 = 24 小時）
- **目的**：跨特徵量級正規化，使其具可比性
- **截斷**：Z-score 在 Bull/Bear Power 中使用前截斷至 [-3, 3]

### 5.3 特徵選擇（v4 — per-target）

**方法**：per-target feature selection（v4 設計）

每個預測 target 有獨立的特徵集，避免跨 target 干擾：

```
1. 基礎特徵集：72 個共用候選特徵
2. 對每個 target 獨立做 ablation study：
   - 逐一加入/移除候選特徵
   - 測量 walk-forward ICIR 變化
3. 只保留對該 target 有正貢獻的特徵
4. 結果：每個 target 獨立的 feature_cols.json
```

**Per-Target 特徵配置**：

| Target | 特徵數 | 額外加入的 OI 特徵 | ICIR |
|--------|--------|-------------------|------|
| `up_move_vol_adj` | 71 | `cg_oi_close_pctchg_8h`、`cg_oi_range_zscore` | 4.04 |
| `down_move_vol_adj` | 70 | `cg_oi_range_pct` | 3.80 |
| `strength_vol_adj` | 69 | （無 — 所有 OI 候選特徵均有害） | 0.18 |

## 6. 機器學習管線

### 6.1 目標變數（v3 — 雙目標架構）

v3 將單一收益率目標拆分為**上行/下行雙目標**，分別預測波動幅度：

```python
# 計算 4h 前瞻收益率
future_return = close.shift(-4) / close - 1

# 雙目標拆分（經 volatility 正規化）
up_move   = max(future_return, 0) / realized_vol   # 上行幅度（≥ 0）
down_move = max(-future_return, 0) / realized_vol   # 下行幅度（≥ 0）

# 方向由非對稱性決定
strength = up_pred - down_pred   # 正 = 看多，負 = 看空
```

**三個訓練 target**：

| Target | 型別 | Active | 說明 |
|--------|------|--------|------|
| `up_move_vol_adj` | ≥ 0 | ✅ | 上行潛力（vol 正規化） |
| `down_move_vol_adj` | ≥ 0 | ✅ | 下行潛力（vol 正規化） |
| `strength_vol_adj` | ±  | ❌ | 淨方向（已停用 — 無信號） |

**為何拆分**：單一收益率目標混合了方向與幅度。拆分後，模型可以獨立評估「上漲有多猛」和「下跌有多猛」，再由差值推導方向。

### 6.2 Walk-Forward 交叉驗證

系統使用嚴格的擴展窗口 walk-forward 驗證，防止資訊洩漏：

```
時間 ──────────────────────────────────────────────────────►

│◄──── 全量訓練資料 (4006 bars) ────────────────────────────►│
│                                                            │
│ [訓練 Fold 1]              │[測試 1]                       │
│ ████████████████████████████│▒▒▒▒▒▒▒                       │
│                                                            │
│ [訓練 Fold 2]                     │[測試 2]                │
│ ████████████████████████████████████│▒▒▒▒▒▒▒               │
│                                                            │
│ [訓練 Fold 3]                             │[測試 3]        │
│ ████████████████████████████████████████████│▒▒▒▒▒▒▒       │
│                                                            │
│ ...（擴展窗口，共 5 折）                                     │
```

**關鍵特性**：
- 訓練總是從第 0 根 bar 開始（擴展窗口）
- 無未來資料洩漏：每折只用測試期之前的資料訓練
- Per-target 特徵選擇在全量資料上做 ICIR ablation
- OOS 預測永遠不用於任何訓練決策

### 6.3 模型架構（v3 — Regime-Conditional）

每個 target 訓練 **4 個 XGBoost 模型**（1 global + 3 regime-specific），推論時加權混合：

```python
# 推論邏輯
global_pred = global_model.predict(x)
regime_pred = regime_model[current_regime].predict(x)

final_pred = (1 - REGIME_BLEND_WEIGHT) * global_pred + REGIME_BLEND_WEIGHT * regime_pred
# REGIME_BLEND_WEIGHT = 0.35
```

**XGBoost 參數**：

```python
XGB_PARAMS = {
    "n_estimators": 500,
    "max_depth": 5,
    "learning_rate": 0.02,
    "subsample": 0.8,
    "colsample_bytree": 0.6,
    "min_child_weight": 3,
    "reg_alpha": 0.05,        # L1 正則化
    "reg_lambda": 0.5,        # L2 正則化
    "early_stopping_rounds": 40
}
```

**Regime 分類**（trailing-only，無前瞻）：
```python
ret_24h = close.pct_change(24)
vol_24h = log_return.rolling(24).std()
vol_pct = vol_24h.expanding(min_periods=168).rank(pct=True)

TRENDING_BULL  = (vol_pct > 0.6) & (ret_24h > +0.5%)
TRENDING_BEAR  = (vol_pct > 0.6) & (ret_24h < -0.5%)
CHOPPY         = 其他
WARMUP         = 前 168 bars（1 週）
```

### 7.2 IndicatorEngine 類別 (`indicator/inference.py`)

```python
class IndicatorEngine:
    """v4 具狀態的預測引擎：regime-conditional dual-target。"""

    def __init__(self):
        # 載入 per-target 模型：{target: {regime: XGBRegressor}}
        self.models: dict[str, dict[str, xgb.XGBRegressor]]
        # 載入 per-target 特徵集
        self.target_feature_cols: dict[str, list[str]]
        # 滾動預測歷史（擴展式）
        self.pred_history = deque(maxlen=500)  # 從 training_stats 暖機

    def predict(self, features: pd.DataFrame) -> pd.DataFrame:
        # 1. 特徵對齊（檢查超集，per-target 子集）
        # 2. Regime 偵測（trailing-only）
        # 3. 每個 bar、每個 target：
        #    global_pred × 0.65 + regime_pred × 0.35
        # 4. strength = up_pred - down_pred
        # 5. direction = strength vs DEADZONE
        # 6. confidence = mag_score（|strength| 擴展百分位）
        # 7. Bull/Bear Power = Coinglass z-scores 均值
        ...
```

**狀態管理**：
- `pred_history`：最近 500 筆 |strength| 的 Deque（供 mag_score 計算）
- 暖機：預載 200 筆歷史 OOS 預測
- 執行緒安全：Flask 路由透過 `threading.Lock()` 保護的共享狀態讀取

### 7.3 方向判定邏輯（v3 — 雙目標非對稱性）

```python
# v1（已棄用）：direction = sign(zscore(pred))  ← z-score 破壞信號
# v2（已棄用）：direction = sign(pred_return) + |pred| > 0.0006
# v3（現行）：  direction = sign(up_pred - down_pred) + |diff| > DEADZONE

STRENGTH_DEADZONE = 0.50

strength = up_pred - down_pred   # 正 = 上行潛力 > 下行潛力

direction = "UP"      若 strength >  STRENGTH_DEADZONE
          = "DOWN"    若 strength < -STRENGTH_DEADZONE
          = "NEUTRAL" 其他（~28% 的 bars）
```

**Deadzone 校準**：
- 0.15（初始）：80% bars 有信號 → 太多噪音，方向頻繁翻轉
- **0.50（現行）**：~72% bars 有信號 → 更穩定，減少假翻轉

### 7.4 信心評分（v3 — 純幅度）

```python
# v1（已棄用）：confidence = rolling z-score 百分位
# v2（已棄用）：confidence = mag_score × regime_score（regime_score 有 overfit）
# v3（現行）：  confidence = mag_score only

mag_score = percentile_rank(|strength|, 所有先前歷史) × 100
confidence = clip(mag_score, 0, 100)
```

- **參考分佈**：擴展歷史分佈（穩定，單調增長）
- **範圍**：0–100
- **解讀**：「|up_pred - down_pred| 相對歷史有多極端？」
- **regime_score 移除原因**：trailing IC 在冷啟動時虛高（訓練/即時資料重疊），OOS 後迅速衰減至 ~0。移除後信心評分更穩健。

### 7.5 強度分級

```python
STRONG_THRESHOLD   = 80.0   # confidence ≥ 80 → Strong
MODERATE_THRESHOLD = 65.0   # confidence ≥ 65 → Moderate
                            # confidence <  65 → Weak
```

**參數演進**：
- v2：Strong ≥ 90、Moderate ≥ 40 → 太少 Strong、太多 Moderate
- v3（初始）：Strong ≥ 90 → 訊號過少
- **v3（現行）**：Strong ≥ 80、Moderate ≥ 65 → 分佈更合理

### 7.6 Bull/Bear Power

由 5 個 Coinglass 衍生品指標合成的情緒分數：

```python
components = [
    +cg_oi_delta_zscore / 3,        # OI 增加 → 偏多
    -cg_funding_close_zscore / 3,    # 高資金費率 → 逆勢偏空
    +cg_taker_delta_zscore / 3,      # 淨買入 → 偏多
    -cg_ls_ratio_zscore / 3,         # 極端多頭 → 逆勢偏空
    +cg_ls_divergence_zscore / 3,    # 背離 → 知情流量
]
bull_bear_power = mean(components).clip(-1, +1)
```

### 9.2 實驗追蹤

#### E1：固定特徵選擇（已完成）

#### E7：Confidence Scoring 審計 + 修復（已完成）

## 12. 模型效能與驗證

### 12.1 當前指標（v4 — dual-target per-target fs）

| 指標 | 數值 | 說明 |
|------|------|------|
| up_move ICIR | **4.04** | 上行幅度預測穩定性（5-fold walk-forward） |
| down_move ICIR | **3.80** | 下行幅度預測穩定性 |
| strength ICIR | +0.18 | 方向強度（已停用此 target） |
| 方向準確率 | **~50%** | 已知限制 — 特徵預測幅度而非方向 |
| STRENGTH_DEADZONE | 0.50 | 信號觸發閾值 |
| Strong 信號佔比 | ~45% of signals | confidence ≥ 80 |
| NEUTRAL 佔比 | ~28% of bars | |strength| < deadzone |

### 12.2 v4 參數一覽

```python
STRENGTH_DEADZONE    = 0.50     # |up_pred - down_pred| < 此值 → NEUTRAL
STRONG_THRESHOLD     = 80.0     # confidence ≥ 80 → Strong
MODERATE_THRESHOLD   = 65.0     # confidence ≥ 65 → Moderate
REGIME_BLEND_WEIGHT  = 0.35     # regime 模型權重（vs global 0.65）
HORIZON_BARS         = 4        # 4h 預測視窗
MIN_MAG_HISTORY      = 30       # mag_score 最低歷史長度
ACTIVE_TARGETS       = ["up_move_vol_adj", "down_move_vol_adj"]
```

### 12.3 Regime 分析

| Regime | 佔比 | 特性 | 解讀 |
|--------|------|------|------|
| CHOPPY | ~59% | 低波動，無明確趨勢 | 模型主力區間 |
| TRENDING_BULL | ~16% | 高波動 + 上漲 | 中等信號 |
| TRENDING_BEAR | ~21% | 高波動 + 下跌 | 中等信號 |
| WARMUP | 前 168 bars | 資料不足 | 不產出 regime 模型 |

### 12.4 防過擬合措施

1. **Per-target 特徵選擇**：每個 target 獨立 ICIR ablation，避免跨 target 干擾
2. **擴展窗口訓練**：每折訓練至測試起始前的所有資料
3. **正則化**：L1（α=0.05）+ L2（λ=0.5）+ 提前停止（40 輪）
4. **子採樣**：80% 列採樣，60% 欄採樣（每棵樹）
5. **Confidence overfit 防護**：移除 regime_score（trailing IC 冷啟動虛高），僅使用 mag_score
6. **OOS 追蹤**：`backtest_oos.py` 累積真正 OOS 資料，定期驗證

## 14. 關鍵設計決策

### 14.1 指標，而非策略

系統刻意定位為**預測指標** — 預測方向與信心，但不產生交易信號。這種分離：
- 避免對回測 P&L 過擬合（會引入存活者偏差）
- 純粹以預測品質（IC、校準性）評估 — 這些指標具泛化能力
- 允許使用者將指標納入自己的決策框架

### 14.2 訂單流 + 衍生品資料（排除傳統技術指標）

特徵集刻意排除傳統技術指標（MACD、EMA、RSI、布林通道）。所有特徵來自：
- **Tick 級訂單流**：CVD、delta ratio、taker 失衡
- **衍生品微結構**：OI、資金費率、清算、多空比
- **跨源背離**：資金費率 × taker 對齊、OI × 價格背離
- **統計矩**：實現波動率、收益偏態/峰度

**理由**：技術指標是價格衍生且廣為人知。訂單流和衍生品資料代表機構參與者的實際持倉與流量 — 較難被套利的 alpha 來源。

### 14.3 1h 原生精度

系統從 15 分鐘遷移至 1 小時 bars，因為：
- Coinglass API 提供原生 1h 資料（vs 30m 時 50% forward-fill）
- NaN 率從 ~50% 降至 ~0.4%
- 特徵來源之間更好的時間對齊
- 4h 視窗（4 bars ahead）在 1h 精度下仍然適當

### 14.4 Per-Target 特徵選擇

不同預測目標需要不同的特徵集：
- **問題**：11 個 OI 特徵幫助 up_move（ICIR +1.65）但傷害 down_move 和 strength
- **解法**：每個 target 獨立做 ICIR ablation，只保留正貢獻特徵
- **結果**：up_move ICIR 3.40→4.04、down_move 2.70→3.80、strength -0.63→+0.18

### 14.5 雙目標非對稱性推導方向

使用 up_pred - down_pred 差值推導方向，而非直接預測收益率：
- **優勢**：模型擅長預測「上行有多猛」和「下行有多猛」（ICIR > 3.8）
- **已知限制**：方向準確率 ~50%（特徵預測波動幅度而非方向）
- **設計選擇**：接受此限制，以 DEADZONE + confidence tier 控制信號品質
- 符合產品目標：「重大走勢的指標」而非「每根 bar 都猜方向」

### 14.6 Confidence 純幅度設計（移除 regime_score）

信心評分僅基於 |strength| 的歷史百分位（mag_score）：
- **被移除的方案**：regime_score = trailing Spearman IC per regime
- **移除原因**：冷啟動時訓練/即時資料重疊，trailing IC 虛高至 0.84，OOS 後迅速衰減至 ~0
- **教訓**：任何依賴 in-sample 重疊資料的 meta-signal 都會 overfit

### 14.7 三服務分離

獨立服務允許：
- 獨立擴展（市場資料是 I/O-bound；指標是 CPU-bound）
- 零停機模型更新（僅替換指標服務）
- 故障隔離（WebSocket 斷線不影響預測）
- 獨立開發週期

## 附錄 B：評估指標詞彙表

| 指標 | 公式 | 解讀 |
|------|------|------|
| **Spearman IC** | rank_corr(pred, actual) | 預測排序品質（-1 至 +1）；>+0.03 具可操作性 |
| **ICIR** | mean(fold_ICs) / std(fold_ICs) | 信號穩定性；>1.0 良好，>2.0 優秀 |
| **方向準確率** | 正確方向數 / 活躍信號數 | UP/DOWN 預測與實際符號匹配的百分比 |
| **校準單調性** | 相鄰 decile 正確排序的百分比 | 更高預測 → 更高實際收益？ |
| **每週正 IC** | IC>0 的週數 / 總週數 | 信號的時間一致性 |
